# TASKS_list — оставшаяся работа по фазам 0–12 и план оркестрации

Задачи покрывают **всё невыполненное** из `SPEC_0-12.md` (пункты `[ ]` и
`[~]`). Каждая задача самодостаточна: агент-исполнитель может работать по ней,
не читая остальных. Ссылки вида `10.1.2` — идентификаторы пунктов SPEC.

Принципы исполнения (обязательны для каждой задачи):
- дисциплина слоёв: математика в `domain`, транспорт в `data`, персист в
  `storage`, оркестрация/IPC в `app`, UI в `frontend`;
- сетевое/тяжёлое — за cargo-фичами; `cargo test --workspace` без фич
  остаётся зелёным;
- секреты только через резолвер (env → `.env` → keyring), не в логи/UI;
- Definition of Done каждой задачи: `cargo fmt --check` +
  `cargo clippy --workspace -- -D warnings` + `cargo test --workspace`
  (+ фиче-гейтед тесты задачи) + для фронта `npm run check && npm run test
  && npm run build`; конвенциональный коммит.

### T1 — HTTP-слой `data::http` `S.4.1–S.4.3`
**Что сделать.** Новая фича `http` в `crates/data`: модуль `data::http` —
тонкая обёртка над `reqwest` (rustls, gzip, тайм-ауты). `HttpClient` с
методом `get_json(url, headers) -> Result<serde_json::Value, DataError>`:
повторы транзиентных сбоев через существующий `data::Backoff`, маппинг
4xx/5xx/сетевых ошибок в `DataError` с корректной `is_retryable`
(429/5xx/timeout — ретраябельны, 4xx — нет). Расширить `data::Method`
вариантами `MoexTradestats`, `MoexFutoi`, `MoexHi2`, `MoexObstats`,
`MoexOrderstats`, `MoexCandles`, `MoexOptions`, `Llm`; `HttpClient` держит
per-method лимит через `RateLimiter`. Транспорт за трейтом (`HttpTransport`),
чтобы оркестрация тестировалась без сети.
**Файлы.** `crates/data/src/http.rs` (новый), `crates/data/src/rate.rs`
(варианты Method), `crates/data/Cargo.toml` (фича `http`).
**Критерии.** Юнит-тесты на фейк-транспорте: сборка URL/заголовков, ретраи
по классам ошибок, соблюдение лимита; `cargo test -p data --features http`
зелёный; дефолтная сборка не тянет `reqwest`.
**Агент:** Sonnet · effort **medium**. Шаблонная инженерия по образцу
существующих `AuthManager`/`Backoff`; риски малы.

## План оркестрации

```
P1 (пользователь: egress + ключи)
│
Волна 1 (параллельно):  T1 http   T2 фикстуры(←P1)   T3 персист   T4 фид   T5 методика MOEX
│                        │         │
Волна 2 (параллельно):  T6 moex(←T1,T2)   T7 llm(←T1,→T3)   T8 storage ALGO
│                        │
Волна 3 (параллельно):  T9 история(←T6)   T11 ipc+фронт ALGO(←T6,T8)   T12 доска(←T6,T5)
│                       T10 загрузчик(←T9)
│
Волна 4:  T13 ревизия(←всё)  →  T14 живой смоук(←P1)  →  T15 упаковка(десктоп)
```

### Правила оркестрации

1. **Одна задача — один агент — одна ветка/PR.** Внутри волны задачи не
   трогают одни и те же файлы, кроме известных стыков: `SCHEMA_VERSION`
   (T8↔T9 — мёржить последовательно), `data/Cargo.toml` (T6↔T7 — фичи
   аддитивны, конфликт тривиален), `mock.ts`/`ipc.ts`/`tauri_app.rs`
   (T10↔T11↔T12 — команды аддитивны).
2. **Контекст агенту**: ссылка на задачу здесь + соответствующие пункты
   `SPEC_0-12.md` + референс-модуль (для транспорта — `data::grpc`/`auth`,
   для IPC — контур Key Activity, для storage — `bars`/`trades`).
3. **Гейт волны**: волна закрыта, когда все её PR зелёные в CI и смёржены;
   T13 запускается только после волны 3.
4. Задачи, требующие сети (T2, T14), не блокируют офлайн-цепочку: T6
   допустимо начинать на `(unverified)`-фикстурах по публичной документации
   ISS с обязательной перепроверкой в T14.
