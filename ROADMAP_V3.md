# ROADMAP V3 — алготрейдер, мультиактивный анализ потоков, «Bot Radar»

Roadmap-предложение новой функциональности поверх текущего терминала (Фазы
0–9 готовы, Фазы 10–12 — доменное ядро MOEX ALGO/Историзация/Опционы, см.
`ROADMAP.md`/`ROADMAP_PHASE_10-12.md`). Это **черновик-предложение** (статус
итерации «proposal» — ничего не реализовано), цель которого — превратить
терминал из «наблюдателя за потоками» в рабочую станцию **алготрейдера**:
портфельный бэктестинг, торговля опционами, межрыночный анализ (акции ·
фьючерсы · облигации · валюта/спот) и — отдельным крупным блоком — детекцию
рыночных роботов и нестандартного поведения участников.

Стиль и дисциплина — как в `ROADMAP_PHASE_10-12.md`: вся математика и логика
обнаружения — в `domain` (чисто, без сети, юнит-тесты на синтетических
данных); сеть/БД — в `data`/`storage` за cargo-фичами; `app` — тонкий IPC;
`frontend` — Svelte 5 + TS, переиспользует существующие компоненты (`Panel`,
`TabBar`, ECharts/Lightweight Charts, `lib/assetClass.ts`) и тёмную тему.

Легенда: `[ ]` — не сделано, `[x]` — сделано, `⏳` — в работе. Идентификаторы
задач (`13.2.3`) стабильны для будущих ссылок (дизайн-бриф, трекер, PR).

> `(verify)` — требует подтверждения по живым данным/документации API
> (формат ответа, доступность поля, лимиты) до реализации соответствующего
> блока — как и в `ROADMAP_PHASE_10-12.md`.

## Принятые решения и ограничения

- **Язык:** roadmap — RU (как остальные), код — EN, UI — RU.
- **Источники данных.** Базовый источник остаётся Finam Trade API (акции,
  фьючерсы, частично облигации). Для FX/спота, полной линейки облигаций и
  части межрыночной аналитики потребуется второй источник — рабочая гипотеза
  `MOEX ISS` (как уже используется в Фазе 10 для ALGOPACK). Везде, где план
  опирается на данные **вне** текущего покрытия Finam, задача помечена
  `(verify)` и/или `(данные)`.
- **Честное ограничение по «Bot Radar» (Фаза 13).** Глубокая, юридически
  значимая детекция спуфинга/лееринга/quote stuffing в индустрии строится на
  **L3-данных уровня сообщений** (insert/modify/cancel с ID заявки и
  таймстампом биржи в микросекундах) — такой ленты у розничного Finam Trade
  API нет (есть лента сделок, снимок стакана по запросу, live-стрим
  `OrderBook`/`Trades`). Поэтому Фаза 13 спроектирована как **двухуровневая**:
  - уровень A — детекторы, которые честно работают на **доступных данных**
    (лента сделок + периодические снимки DOM + бары), и дают приближённые,
    но полезные сигналы;
  - уровень B — детекторы, которые **корректны только при L3/полной книге
    ордеров** — помечены явно и реализуются как опциональный модуль за
    отдельным источником данных (например, если в будущем появится доступ к
    более полной ленте биржи или к историческим L3-датасетам для бэктеста
    методики). Они не блокируют уровень A.
  Эта честность — намеренная: лучше калиброванный «вероятностный радар» на
  реальных данных, чем имитация выдуманных метрик.
- **Производительность.** Детекторы роботов и межрыночная статистика — это
  потоковые/оконные вычисления над тиковыми данными; всё, что не помещается в
  `domain` как чистая функция над срезом, выносится в фоновый `app`-таск
  (как `IngestService`) с собственным тактом, не блокирующим UI.

## Фаза 13 — «Bot Radar»: детекция рыночных роботов и нестандартного поведения

Идея фазы: не один индикатор, а **набор независимых детекторов** (каждый —
дешёвая чистая функция в `domain::botradar`), результаты которых объединяются
в **профиль поведения** по инструменту/окну времени с весами и confidence —
аналог того, как уже работает `domain::keyactivity` (правила → свод). Каждый
детектор работает самостоятельно и тестируется на синтетических фикстурах
(сконструированный «образцовый» паттерн → детектор должен его поймать; чистый
случайный шум → детектор не должен ложно срабатывать чаще базовой частоты).

### 13.0 — Инфраструктура
- [ ] `13.0.1` Крейт-модуль `domain::botradar`: общие типы `Tape` (срез ленты
  сделок), `BookSnapshotSeries` (серия снимков DOM с таймстампами — то, что
  реально можно набрать поллингом `order_book`), `Signal { kind, ts, symbol,
  score: f64 (0..1), evidence: Vec<Evidence> }`.
- [ ] `13.0.2` `storage`: таблица `book_snapshots` (символ, ts, биды/аски
  JSON/колонки) — ретеншн настраиваемый (короткий, т.к. объём большой);
  пишется планировщиком поллинга DOM (переиспользует `BatchCursor`, как
  ингест баров).
- [ ] `13.0.3` Скоринг и объединение сигналов в `RobotProfile` (см. `13.12`).

### 13.1 — Уровень A: детекторы на доступных данных (лента + периодический DOM)

**13.1.1 — Расширенная детекция айсбергов (углубление существующего).**
Текущий `domain::delta::robots::detect_icebergs` ловит «доливки» сравнением
объёма с book-уровнем. Расширение: статистика **времени между доливками**
(refill latency) — у алгоритмических айсбергов оно на порядок стабильнее
(низкая дисперсия), чем у органического потока заявок на том же уровне;
плюс оценка **скрытого суммарного объёма** через экспоненциальное сглаживание
серии доливок (классический estimator «hidden iceberg size»).
- [ ] Реализовать `IcebergProfile { refill_count, refill_interval_cv,
  estimated_hidden_qty }`; `refill_interval_cv < порог` → высокая уверенность
  «это алгоритм», не ручная докупка.

**13.1.2 — Momentum ignition / охота за стопами (stop-hunt).**
Сигнатура: резкий импульсный выброс цены за пределы недавнего диапазона
(swing high/low либо VWAP-полосы из `domain::algo::tradestats`) на
**аномально малом объёме** относительно типичного объёма такого ценового
шага, за которым следует быстрый возврат (mean-reversion) в течение N баров.
Интуиция: настоящий информированный поток двигает цену *пропорционально*
объёму; импульс на «слишком дешёвом» объёме с быстрым откатом — типичная
сигнатура алгоритма, провоцирующего срабатывание чужих стоп-заявок.
- [ ] `detect_momentum_ignition(bars, vwap_band, lookback)` — порог
  «цена/объём» через z-score исторического отношения хода цены к объёму;
  подтверждение отката (reversion) в окне `K` баров.

**13.1.3 — «Пинг-понг» принты / признаки wash-подобной активности.**
На анонимной ленте нельзя доказать wash trading (нужны ID счетов), но можно
детектировать **статистически маловероятный паттерн принтов**: серия
сделок одинаковой цены и размера, чередующихся buy/sell с нулевым чистым
сдвигом цены и объёма стакана, частота которых значимо превышает базовую
(оценённую Пуассоном по недавней активности тикера). Маркируется как
*«паттерн, похожий на круговые сделки»*, не как утверждение о манипуляции.
- [ ] `detect_ping_pong_prints(tape, window)` — тест частоты «zero-impact
  round trips» против пуассоновского бейзлайна.

**13.1.4 — «Heartbeat»: периодические (таймер-driven) роботы.**
Многие простые торговые боты реагируют по таймеру опроса (poll loop), а не
событийно — характерная сигнатура: пик в спектре межсделочных интервалов на
фиксированной частоте (БПФ или автокорреляция временного ряда таймстампов
сделок данного участника-прокси — фильтруем по цене/размеру как proxy
«одного и того же источника потока»). Органический/событийный поток такого
пика не даёт (распределение интервалов ближе к экспоненциальному).
- [ ] `detect_heartbeat(trade_timestamps)` — автокорреляция межсделочных
  интервалов + поиск доминирующего лага; порог по отношению пика к шуму
  (аналог SNR). Забавная деталь: тот же метод детектирует и наш собственный
  поллинг-движок при доступе к его публичному следу — полезный sanity-check.

**13.1.5 — Энтропия и «круглость» размеров заявок/сделок.**
Органический розничный + институциональный поток имеет разнообразное,
скошенное распределение размеров; наивные роботы часто бьют **фиксированными
или «круглыми» лотами** (ровно 100/500/1000 и т.п.). Метрика: энтропия
Шеннона распределения размеров за окно + доля «круглых» размеров против
исторической нормы инструмента (z-score). Низкая энтропия + всплеск доли
круглых лотов = сигнатура алгоритмического источника.
- [ ] `order_size_entropy(sizes) -> f64`, `round_lot_share(sizes) -> f64`,
  сравнение с roll-баром собственной истории инструмента (self-baseline —
  не нужны размеченные данные).

**13.1.6 — HHI объёма по ценовым уровням (переиспользование `algo::hi2`).**
`domain::algo::hi2` уже считает индекс Херфиндаля **концентрации оборота
между инструментами**. Та же математика, применённая **внутри инструмента к
распределению объёма по ценовым уровням ленты**, ловит «стенки» —
аномальную концентрацию объёма на 1–2 уровнях, типичную для алгоритмической
защиты уровня или айсберг-кластера.
- [ ] `price_level_concentration(tape, tick_size) -> Hi2Like` — тонкая
  обёртка над существующей функцией HHI из `algo::hi2` с другим измерением
  агрегации (переиспользование, не дублирование математики).

**13.1.7 — VPIN (Volume-Synchronized Probability of Informed Trading).**
Известная микроструктурная метрика «токсичности» потока — оценивает долю
объёма, инициированного информированной стороной, через объёмные бакеты и
дисбаланс buy/sell-агрессии (используем уже размеченную сторону агрессора из
ленты, как в `domain::delta`). Полезна как **ранний индикатор** повышенного
алгоритмического/информированного присутствия перед резкими движениями —
своего рода предвестник «flash»-эпизодов.
- [ ] `domain::botradar::vpin` — объёмные бакеты фиксированного размера,
  скользящее окно бакетов, нормированный дисбаланс; тесты на синтетике
  (искусственный «токсичный» период против спокойного).

**13.1.8 — Межинструментное «теневое копирование» (shadowing / lead-lag).**
Для связанных инструментов (акция ↔ её фьючерс; индекс ↔ компоненты сектора)
ищем устойчивый лаг, при котором поток сделок одного инструмента предсказывает
поток другого с высокой кросс-корреляцией на коротком сдвиге (сотни мс — units
секунд, в пределах разрешения снимков/тиков, которое реально доступно).
Стабильный ненулевой лаг = вероятный алгоритм-арбитражёр/шэдоуинг, а не
совпадение. Переиспользует матрицу перетоков `domain::metrics::crossasset`
как источник пар инструментов для проверки.
- [ ] `cross_instrument_lag(tape_a, tape_b, max_lag) -> LagCorrelation` —
  кросс-корреляция событийных рядов (агрегированных в равные тайм-бакеты) по
  сетке лагов; репортится лучший лаг + сила корреляции.

**13.1.9 — Форма стакана: самореференсная аномалия.**
Из периодических снимков DOM (`13.0.2`) строим «профиль глубины» (объём по
относительным от mid уровням) и сравниваем текущий снимок с собственной
исторической нормой инструмента в это же время сессии (не нужен размеченный
эталон по рынку — каждый тикер сравнивается сам с собой). Внезапная
симметричная «стена» далеко от тача, которая то появляется, то исчезает в
такт ценовому приближению — классическая сигнатура layering-подобного
поведения на доступном нам уровне данных (приближение уровня B).
- [ ] `book_shape_anomaly(snapshots, history_baseline) -> f64` — нормализация
  профиля глубины + расстояние (например, Jensen-Shannon) до бейзлайна
  того же времени суток.

### 13.2 — Уровень B: детекторы, требующие L3/полной книги ордеров (опционально)

Помечены `(verify, требует L3-источник)` — реализуются только если появится
подходящий источник (биржевой фид с ID заявок, исторический L3-датасет для
калибровки). Дизайн закладывается заранее, чтобы не блокировать API.

- [ ] `13.2.1` `(verify)` Истинный спуфинг/лееринг: order-to-trade ratio по
  заявке, доля отменённых заявок «на подходе цены», синхронные отмены на
  нескольких уровнях одним источником.
- [ ] `13.2.2` `(verify)` Quote stuffing: всплеск частоты insert/cancel
  сообщений без сопутствующего роста объёма/волатильности (требует
  message-level ленты, не снимков).
- [ ] `13.2.3` `(verify)` Pinging/sniffing: серии минимальных по размеру
  заявок с низким fill-rate, синхронизированные с последующими крупными
  исполнениями другого участника.

### 13.3 — Таксономия, скоринг и «отпечатки» поведения

- [ ] `13.3.1` **Кластеризация поведенческих отпечатков.** Для каждого
  «эпизода» активности (последовательность сделок/доливок одного
  ценового кластера за короткое окно) строим фиче-вектор: средний размер,
  CV интервала, дистанция от mid на момент появления, энтропия размеров,
  присутствие heartbeat-пика. Кластеризация (DBSCAN, без внешних
  ML-зависимостей — реализация на чистом Rust) выявляет повторяющиеся
  «семьи» поведения на инструменте за период — практический способ увидеть
  «тут работает один и тот же алгоритм» **без доступа к ID счёта**.
- [ ] `13.3.2` **Таксономия архетипов** — `RobotArchetype`: `MarketMaker`
  (двусторонние тонкие котировки у тача, низкая удерживаемая позиция),
  `IcebergExecutor`, `MomentumIgnition`, `StopHunter`, `LikelySpoofer`
  (только уровень B), `QuoteStuffer` (уровень B), `Pinger` (уровень B),
  `CrossAssetArbitrageur` (по `13.1.8`), `Unclassified`. Правила
  сопоставления сигналов → архетип с confidence — стиль как в
  `domain::keyactivity` (типизированные правила, не ML «чёрный ящик»).
- [ ] `13.3.3` `RobotProfile` на инструмент/окно: список активных сигналов +
  доминирующий архетип + confidence + «доказательная база» (ссылки на
  конкретные сделки/бары для drill-down в UI).

### 13.4 — Storage / App / Frontend

- [ ] `13.4.1` `storage`: таблица `robot_signals` (символ, ts, kind, score,
  evidence JSON) — ингест из фонового скан-таска.
- [ ] `13.4.2` `app::botradar` (фича `botradar`): фоновый таск, который по
  такту прогоняет уровень-A детекторы по вотчлисту на свежих данных (как
  `IngestService`), пишет сигналы, шлёт live-событие `robot:signal`
  (аналог `fill:tick`/`trade:tick`).
- [ ] `13.4.3` IPC: `robot_profile(symbol, fromTs, toTs)`,
  `robot_signals_scan(symbol, fromTs, toTs, config)` (on-demand прогон, как
  уже есть `robot_scan` для текущих 3 детекторов — это естественное
  расширение существующей команды, не новая подсистема).
- [ ] `13.4.4` **Frontend — вкладка/панель «Bot Radar».** Новая вкладка в
  `TabBar` (рядом с Delta/Trade/Backtest): тепловая карта тикеров по
  суммарному score за сессию (ECharts heatmap, как в Фазе 4), лента
  сигналов в реальном времени (как `TimeSales`, но для `robot:signal`),
  карточка `RobotProfile` с архетипом/confidence и drill-down (клик по
  сигналу подсвечивает соответствующие сделки на `DeltaChart`/свечах —

  тёмная тема, `Panel.svelte`, русские подписи; цветовая кодировка severity
  (низкая/средняя/высокая) — новая, но согласованная палитра с
  `lib/assetClass.ts`.
- [ ] `13.4.5` Интеграция с `domain::keyactivity`: сигналы Bot Radar — новый
  тип «области» в правилах Key Activity (наравне с ценой/объёмом), попадают
  в LLM-промпт-свод (Фаза 10) как дополнительный контекст для итогового
  резюме сессии.

### 13.5 — Тесты
- [ ] `13.5.1` Каждый детектор — синтетические фикстуры «образцовый паттерн

## Фаза 14 — Мультиактивный бэктестинг, оптимизация, издержки исполнения

- [ ] `14.1` **Портфельный бэктестер.** Расширение `domain::backtest`:
  синхронный движок по нескольким символам одновременно, неттинг позиций,
  маржа/риск на уровне портфеля, единая кривая капитала.
- [ ] `14.2` **Walk-forward анализ.** Скользящие окна train/test, метрика
  деградации out-of-sample (отношение Sharpe test/train).
- [ ] `14.3` **Оптимизация параметров.** Grid search + случайный поиск
  (Rust, без внешних зависимостей) с защитой от переобучения через `14.2`;
  опционально — простой генетический алгоритм для больших пространств
  параметров.
- [ ] `14.4` **Monte Carlo / bootstrap по сделкам.** Ресэмплинг
  последовательности сделок → распределение итоговых метрик (доверительные
  интервалы P&L/просадки), оценка risk-of-ruin.
- [ ] `14.5` **Модель проскальзывания и market impact.** Линейная и
  «квадратный корень от объёма» модели импакта, калибруемые по
  историческому спреду/объёму инструмента (используют уже посчитанный
  footprint из `domain::delta`).
- [ ] `14.6` **TCA (Transaction Cost Analysis).** Сравнение факта исполнения
  симулятора с бенчмарками (arrival price, VWAP периода); отчёт по
  издержкам на сделку/стратегию.
- [ ] `14.7` **Мониторинг распада альфы.** Скользящий Sharpe/win-rate
  стратегии в paper-режиме; алёрт при устойчивой деградации
  («strategy graveyard»).
- [ ] `14.8` Frontend: портфельный режим Бэктестера, heatmap по сетке двух
  параметров, fan-chart Monte Carlo (ECharts), панель walk-forward.
- [ ] `14.9` Тесты: портфельный неттинг, walk-forward на синтетических рядах
  с известным сдвигом режима, bootstrap — статистические свойства на
  контролируемых данных.

## Фаза 15 — Опционы 2.0: поток, дилерское позиционирование, бэктест стратегий

Опирается на доменное ядро опционов Фазы 12 (ценообразование, греки, улыбки,
конструктор стратегий) — уже готово; здесь добавляется **поток/сентимент** и
**бэктест поверх исторической доски**.

- [ ] `15.1` **Options Flow.** Необычная активность: объём/открытый интерес
  ratio по страйку, всплеск премиального оборота, **sweep-детекция**
  (серия мелких сделок по нескольким страйкам/сериям в короткое окно —
  сигнатура алгоритмического исполнения крупного направленного ордера через
  опционы).
- [ ] `15.2` **Gamma Exposure (GEX) и дилерское позиционирование.** Оценка
  чистой гаммы маркет-мейкеров по доске при стандартных допущениях
  (дилер short call / long put или наоборот по конвенции), уровень
  «gamma flip», эвристика влияния на реализованную волатильность спота.
- [ ] `15.3` **Sentiment-индикаторы.** Put/Call ratio (объём и OI), term
  structure skew, 25-delta risk reversal — на базе уже откалиброванных
  улыбок (`domain::options::smile`).
- [ ] `15.4` **Бэктестер опционных стратегий.** Прогон шаблонов стратегий
  (`domain::options::strategy`) по исторической доске с разложением P&L по
  грекам (delta/gamma/theta/vega attribution) — переиспользует движок Фазы
  14.
- [ ] `15.5` **Vol surface во времени.** Анимация/таймлайн поверхности
  волатильности, событийная разметка (экспирация, дивотсечки) —
  «vol crush» после экспираций как отдельный сигнал.
- [ ] `15.6` Frontend: расширение вкладки «Опционы» — лента Flow, GEX-чарт,
  панель P&L-аттрибуции бэктеста.
- [ ] `15.7` `(verify)` Доступность исторической опционной доски (объём,
  OI, по страйкам/сериям) через MOEX ISS — формат и глубина истории.
- [ ] `15.8` Тесты: sweep-детекция и GEX на синтетических досках с известным
  ответом.

## Фаза 16 — Межрыночные потоки и ротация капитала

Расширение существующего `domain::metrics::crossasset` (Фаза 6) от
«снимка долей» к **динамическому межрыночному анализу**.

- [ ] `16.1` **Risk-on/Risk-off композит.** Индекс на основе относительной
  силы акций vs защитных активов (облигации/денежный рынок) — переиспользует
  RRG-математику (Фаза 4) на уровне классов активов, а не секторов.
- [ ] `16.2` **Корреляции и скользящая бета.** Матрица корреляций
  между классами активов/секторами/инструментами на скользящем окне;
  heatmap + детекция «разрыва корреляции» (regime change в со-движении
  рынков — полезно и как фильтр для парного трейдинга Фазы 18).
- [ ] `16.3` **Базис фьючерс-спот и календарные спреды.** Контанго/
  бэквордация по срокам контрактов одного базового актива; сигнал на
  аномальное расширение базиса (cash-and-carry возможность).
- [ ] `16.4` **Lead-lag между классами активов.** Переиспользует
  кросс-корреляционную инфраструктуру `13.1.8` на агрегированных потоках
  классов активов вместо пары инструментов — «куда смотреть дальше», если
  фьючерсы уже двинулись, а акции/облигации ещё нет.
- [ ] `16.5` **Сезонность.** Профиль оборота/потоков по времени суток/дню
  недели/месяцу — на уровне секторов и классов активов.
- [ ] `16.6` Frontend: новая панель «Межрыночный анализ» во вкладке «Сумма
  всех» (или отдельная вкладка) — корреляционная heatmap, базис-чарт,
  risk-on/off gauge (переиспользует `TotalTurnoverGauge`-паттерн).
- [ ] `16.7` Тесты.

## Фаза 18 — Статистика и режимы рынка

- [ ] `18.1` **Детекция режима рынка.** Простая HMM или байесовский

## Порядок и зависимости

1. **Фаза 13 (Bot Radar), уровень A** не зависит от остальных новых фаз —
   опирается только на существующие `domain::delta`/`algo::hi2` и доступную
   ленту/DOM. Может разрабатываться первой и параллельно со всем остальным.
2. **Фаза 14** (портфельный бэктест/TCA/Monte Carlo) — основа для Фазы 15.4
   (опционный бэктест) и Фазы 18.3 (парный трейдинг как стратегия) — делать
   до них.
3. **Фаза 15** опирается на готовое доменное ядро опционов Фазы 12 (уже
   реализовано) + Фазу 14 (движок бэктеста) + `(verify)` 15.7 по доске.
4. **Фазы 16–17** расширяют существующий `crossasset`/`sector`/`rrg`/
   `YieldCurve` — независимы друг от друга, но 17 требует решения по
   источнику FX/расширенных облигаций (`17.5`, та же инфраструктура, что
   `(verify)` MOEX в Фазе 10).
5. **Фаза 18** независима математически, но `18.1` (режим рынка) и `18.2`
   (волатильность) полезны как входы для Фазы 19 (риск) и Фазы 13 (нормировка
   порогов детекторов по текущему режиму волатильности — снижает
   ложные срабатывания на высоковолатильных сессиях).
6. **Фаза 19** опирается на симулятор Фазы 9 (готов) и выигрывает от Фазы 18.2.
7. **Фаза 20** — сквозная инфраструктура, делается по мере готовности
   остальных фаз (уведомления и SDK осмысленны только когда есть что
   уведомлять/подключать).

## Дисциплина и качество (как в проекте)

- Вся математика/детекторы/правила — в `domain`, чисто, без сети, юнит-тесты
  на синтетических фикстурах с явной целью (паттерн ловится / шум не
  ловится чаще целевого false-positive rate).
- Сетевое/тяжёлое — за cargo-фичами (`http`, `moex`, `notify`, `botradar`,
  ...); кросс-платформенный CI остаётся зелёным без них.
- Секреты (Telegram-токен, доп. API-ключи) — только через существующий
  резолвер (env → `.env` → keyring), не в localStorage и не в логах.
- Каждый PR: `cargo fmt --check`, `cargo clippy -D warnings`, `cargo test
  --workspace`, `npm run check`, `npm run test`, `npm run build` (фронтовая
  CI-проверка добавлена в текущей итерации, см. PR с frontend-тестами).
- Коммиты — конвенциональные (`feat:`/`fix:`...), как в истории репозитория.
